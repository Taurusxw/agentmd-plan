# AGENTS.md

> 全局 Codex 工作规则。
> 版本：27.7.0
> 定版日期：2026-07-18
> 目标：用精简、始终在场的纲要约束所有任务；详细执行规则由 `seer-codex-rules` 按需加载，兼顾全面遵守、低 token 和可验证性。

---

# 1. 核心原则

1. 先理解和查证，再判断或修改。
2. 优先沿用项目现有架构、目录、命名、工具和文档体系。
3. 改动保持小而完整，不做无关重构或机械留痕。
4. 所有持久改动必须可复查、可验证、可回溯、可交接。
5. 任务按风险和规模分级，短任务轻量闭环，长任务完整留痕。
6. 避免单文件多职责、重复规则、同义文档和浅层抽象。
7. 保留用户已有修改，不擅自覆盖、回滚、删除或迁移。
8. 不把文字规则当成安全边界；高风险操作仍依赖权限、sandbox、approval、hooks、测试、CI 或人工确认。
9. 验收用于证明当前目标完成，不得无限追加门槛或制造新目标。
10. 最终回答必须诚实说明改动、验证、未覆盖项和风险。

---

# 2. 指令与规则位置

1. 遵守系统、开发者、用户、全局 `AGENTS.md`、项目和子目录规则的有效指令；越靠近目标目录的项目规则越具体。
2. 全局 `AGENTS.md` 只保留通用纲要、强制门禁和安全底线。
3. 项目 `AGENTS.md` 保存项目目标、技术栈、命令、目录、完成标准和风险边界。
4. 子目录 `AGENTS.md` 保存局部模块规则。
5. 可重复、复杂、分支较多的执行流程进入 Skill；项目知识进入项目文档；硬阻断进入工具或权限系统。
6. 除非用户明确要求临时覆盖，不主动创建 `AGENTS.override.md`。
7. 如果规则来源冲突，按有效指令优先级执行，并在必要时说明冲突和取舍。

---

# 3. 任务分级

| 等级 | 典型场景 | 默认闭环 |
|---|---|---|
| L0 只读分析 | 解释、审查、调研、定位，不修改文件 | 读取必要证据，输出结论；不写开发留痕 |
| L1 微小变更 | 单点修复、文案或配置小改，无公开行为变化 | 最小修改、直接验证；默认不新增 round |
| L2 常规开发 | 小功能、缺陷修复、用户可见变化、1-3 个模块 | 更新受影响代码/文档，运行针对性测试，按项目规则留痕 |
| L3 重要变更 | API、数据、权限、安全、依赖、部署、架构或跨模块重构 | 先计划和评估影响，记录决策与风险，运行更完整验证 |
| L4 阶段或发布 | 多日工作、迁移、审计、发布、多人交接 | 使用 phase/release 记录计划、设计、验收、复查和发布风险 |

出现公开接口、数据结构、鉴权、安全、部署、依赖、兼容性、难回滚、跨层、三模块以上影响，或重复补丁热点同时出现职责、接口、重复逻辑漂移时，至少升级到 L3；持续多日、多轮验收或需要交接时升级到 L4。只读请求保持 L0，不擅自修改文件。

---

# 4. 强制 Skill 门禁

1. 任务明确匹配任何现有 Skill 时，必须先完整读取其 `SKILL.md`，再按需读取直接路由的 reference、脚本或模板。
2. 任何会修改代码、配置、文档或规则的开发任务，开始和结束检查都必须读取并遵守 `<codex-home>/skills/seer-codex-rules/SKILL.md`。
3. 普通开发使用轻量 compliance 路径：读取 `SKILL.md`、任务分级 reference 和一个当前产物所需的治理 reference；不机械加载全部规则。
4. 涉及 `AGENTS.md`、项目规则、工作流、任务分级、文档治理、进度留痕、版本号、规则迁移或 Skill 维护时，必须运行该 Skill 规定的 guardrail，并按需读取完整治理 reference。
5. 最终回答必须说明 Skill 使用、关键 reference 和未覆盖风险；普通任务保持一行，规则或发布任务按风险展开。
6. 如果强制 Skill 缺失、无法读取或校验失败，不得把规则敏感或文件修改任务表述为已合规完成，应停止相关写入或明确报告阻塞。
7. Skill 提供详细执行方法，但不能覆盖更高优先级指令，也不能代替权限、安全确认或真实验证。
8. OpenAI、Codex、API、模型、工具、技术版本、法规、价格和云服务等可变事实，优先查官方或当前来源，不把猜测写成规则。

---

# 5. 执行与修改底线

1. 先搜索代码、类型、配置、测试和文档；文件和文本发现优先使用 `rg` / `rg --files`。
2. 需求边界影响数据、权限、破坏性操作、部署或关键体验时，先澄清；安全且可逆的假设可以执行，但要说明。
3. 优先复用现有模块、成熟库、官方示例和稳定工具；新增依赖或抽象必须有明确收益。
4. 修改只覆盖完成目标所需范围；不顺手清理无关代码，不隐藏错误，不把临时 hack 当长期方案。
5. 模块追求小接口、深实现和清晰职责；拆分不得制造无意义的传递包装。
6. 修改后检查引用、导入、路径、命名、类型、测试、文档和生成物是否一致。
7. 删除、重命名、迁移、历史改写或其他破坏性操作前，搜索影响并取得必要确认。
8. 发现用户未提交或非本轮修改时，默认保留并与其协作；无法兼容时再请求方向。
9. 同一生产文件或模块连续成为补丁热点，或新改动会增加独立职责、接口宽度、重复逻辑时，必须按 `seer-codex-rules` 执行架构漂移检查；行数只作信号，不作为强制拆分依据。

---

# 6. 文档、留痕与版本

1. 文档按实际影响更新，不为模板机械创建完整文档树，不创建同义或重复长期文档。
2. L0 不写开发留痕；L1 默认不新增 round；L2 在项目已有进度体系时更新当前 round；L3 记录影响、决策、验证和风险；L4 使用 phase 或 release。
3. 同一目标连续推进优先更新已有 round；只有新目标、新阶段、阻塞解除、验收反馈或风险升级才新增记录。
4. round 编号按日期重置，同一天独立任务才递增；容量阈值只能改变记录位置，不能成为跳过必要留痕的理由。
5. `rounds/` 超过 30 个直接文件时按月份整理，超过 100 个时归档或总结；同一主题连续 5-8 个 round 时升级为 phase。
6. `docs/PROGRESS.md` 只保留状态、近期进展、下一步、风险和详细记录入口。
7. 全局 `AGENTS.md` 目标为 120-220 非空行，达到 300 非空行或 16 KiB 时停止堆叠详细规则，优先迁移到 Skill 或项目规则。
8. 详细文档触发表、命名、模板、阈值和归档算法以 `seer-codex-rules` 的 `documentation-governance.md` 为准。

`AGENTS.md` 使用 `MAJOR.MINOR.PATCH`：

| 变更 | 版本动作 |
|---|---|
| 只讨论、研究、备份或未修改文件 | 不推进 |
| 纯错字、格式、链接或不改变行为的澄清 | PATCH |
| 新增或实质优化长期规则、验证、Skill 路由或治理能力 | MINOR |
| 改变全局工作模型、指令边界或兼容方式 | MAJOR |

修改版本化规则文件时自动判断最高影响级别、更新日期并说明依据；详细历史依赖正式 artifact、状态清单、备份或提交记录，不在全局文件内堆 changelog。

---

# 7. 验证与验收收束

1. 运行能覆盖本次改动的最小可信验证：目标测试、静态检查、lint、typecheck、build、复现命令或实际渲染。
2. L1 使用直接验证；L2 使用针对性测试并复查 diff；L3/L4 按风险增加集成、迁移、接口、安全、回归或发布检查。
3. 前端、图像、PDF、幻灯片、文档排版、游戏和可视化必须实际渲染或截图检查。
4. 命令退出码为零不是唯一证据；必须确认检查覆盖了改动行为。
5. 无法验证时，说明原因、未覆盖范围、应运行的命令和残留风险，不得宣称已经通过。
6. 验收标准通过后立即收尾；外部登录、权限、风控、速率限制或不可用环境默认记录为残留风险，不自动扩张当前目标。
7. 新发现仅在阻塞原始目标或重大安全边界时进入当前任务；流程加固和可选改进进入后续项。

---

# 8. 完成检查

结束前必须确认并说明：

1. 完成了什么，哪些文件或行为发生变化。
2. 运行了什么验证，验证覆盖了什么。
3. 文档、round/phase/release、索引和版本是否需要更新。
4. 是否保留用户修改，是否存在重复文档、未完成事项或风险。
5. 使用了哪些 Skills 和关键 references，还有哪些未覆盖风险。

短任务保持简洁；L2 以上按改动、文档、测试、留痕、风险和下一步组织输出，但不得让输出模板反过来拖慢任务完成。
